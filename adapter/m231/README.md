# Адаптация отчёта `m231` под новые требования

Пакет предназначен для внесения правок в существующее решение заказчика (Spring Boot + Thymeleaf + jQuery + Chart.js + DataTables + Tom Select). Файлы из `adapter/m231/` можно скопировать в проект заказчика с заменой одноимённых.

## Что меняется по сравнению с исходной версией

1. Один большой график делится на два:
   - **Динамика регистрации и решения ЕИ** — stacked-бары «Поступившие» (ЕИ с МИ / без МИ) по дате регистрации и «Решённые» (с МИ / без МИ) по дате решения, плюс линия «Ср. длительность решения», считаемая **только по ЕИ, решённым в этот период**.
   - **Открытые ЕИ по срокам решения (снимок)** — stacked-гистограмма открытых ЕИ по фиксированным бакетам `4ч, 12ч, 24ч, 48ч, 72ч, >72ч` + мини-плашка «Открытых ЕИ N, в т.ч. с МИ M».
2. Убраны линии «с МИ / ППР / Проблемами» (по факту всегда 0 для последних двух, а «с МИ» перенесён в стек).
3. Добавлен фильтр «Срок решения» (multi-select с Tom Select).
4. В SQL добавлена колонка **`Дата решения`** = `FROM_UNIXTIME(COALESCE(a.TIME5, a.CLOSETIME))`.
5. Расчёт «решённый/открытый» унифицирован через `COALESCE(a.TIME5, a.CLOSETIME)` — убрана рассинхронизация между `Кол-во решенных ЕИ` и `Открыт`.
6. **Закрыта SQL-инъекция** в `ReportController.getReportData` — все фильтры переведены на параметризованные запросы (`?` + `params.toArray()`).
7. Формат оси X при `unit=week` — ISO год-неделя `YYYY wWW` (например, `2026 w01`).
8. Чистые функции вынесены в `m231-core.js` и покрыты unit-тестами на Node.

## Куда что класть в проекте заказчика

| Файл из `adapter/m231/` | Путь у заказчика |
| --- | --- |
| `ReportController.java` | `src/main/java/.../ReportController.java` |
| `templates/m231.html` | `src/main/resources/templates/m231.html` |
| `static/css/m231.css` | `src/main/resources/static/css/m231.css` |
| `static/js/m231-core.js` | `src/main/resources/static/js/m231-core.js` (новый) |
| `static/js/m231.js` | `src/main/resources/static/js/m231.js` |

В `m231.html` важно подключить **в таком порядке**:

```html
<script src="/js/m231-core.js"></script>
<script src="/js/m231.js"></script>
```

## Запуск тестов и превью

```bash
# unit-тесты чистых функций (ISO-неделя, бакеты, агрегация)
node --test adapter/m231/test/core.test.mjs

# генерация моковой фикстуры (использует frontend/public/data.json)
node adapter/m231/test/build-mock.mjs

# локальный превью без Spring — подменяет fetch('/reports/data231') на data231.mock.json
cd adapter/m231 && python3 -m http.server 5180 --bind 127.0.0.1
# открыть: http://localhost:5180/test/preview.html
```

## Замечания по безопасности и качеству

- **SQL-инъекции**: в исходном коде WHERE формировался конкатенацией. Теперь все пользовательские параметры уходят через `jdbcTemplate.queryForList(sql, params.toArray())`.
- Типы данных фильтров приведены (`durationFrom/To` — `Double.parseDouble`), `dateFrom/To` сравниваются через `DATE(t.\`Дата регистрации\`)`.
- `clientType` как мульти-значение в текущем контроллере передаётся одной строкой (как было и раньше). Если заказчик хочет полноценный multi-select для `clientType`, это отдельная небольшая доработка: `@RequestParam(required = false) List<String> clientType` + `IN (...)` с плейсхолдерами.

## Логика, которую стоит прояснить на стороне заказчика

- `unit=halfyear` оставлен, хотя это нестандартный период — при необходимости уберите опцию из `<select id="timeUnitFilter">`.
- Кнопка «Сбросить» возвращает фильтр «Тип клиента» на `Мобильный` (как в исходной версии). Если нужно возвращать «все», удалите блок `setValue(["Мобильный"])` из `resetFilters()`.
